Old-Check Format of Import File

This topic introduces lists requirements for data files that you can import into Enterprise SMS Web. It includes two sections: one for file name and format; another for file contents.

File Name and Format

Data files (for example, containing field names and values for real property inventory items) need to be formatted as specified below:

File Name

The file to be imported must be named with the following concatenated together:

  • schema name
  • dot (period)
  • table name

For example: "SMS.Site". The schema and table names specify the destination table for the data.

This naming convention means that the file will have neither the extension .csv nor the extension .txt even though it must be a text file and must contain comma-separated values.

File Format

Text File

The data file to be imported must be a text file. However, it will not have a standard text file extension because the file must be named with the destination schema and table name. (See File Name for exact details).

Comma-Separated Values

The data file to be imported must contain comma-separated values. Only values are transferred; the file should not contain names of the properties that the values correspond with. This means that you need to know the exact structure of the target data table, in order to list the values in the right order, without transmitting either too many or too few values for each record.

In the data file, commas are the separator between values. A carriage return or line break is the separator between records. For example, if importing SMS.Site, each new row is the data for another site.

No Header Row

The data file must NOT have a header row; it needs to start with the first data item.

 

File Contents

To determine what and how many values to send in what order, you need to know the structure of the destination table. Refer to the data dictionary for the relevant domain, if one exists.
