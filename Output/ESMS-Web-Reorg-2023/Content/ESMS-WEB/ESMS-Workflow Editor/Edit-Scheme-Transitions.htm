Edit Scheme Transitions

To view or edit details about a transition, activate (with a double-click) anywhere on the relevant connecting line in the diagram; the "Transition" popup will appear with the transition's details. The "Transitions" popup is where you design in the conditions or permissions needed to transition to the next state in the sequence. You may need to use parameters for some of these.

Transition Attributes

In the field descriptions below, Required items are those required by the software. Your process may include items that are Optional to the software, but required for your workflow.

  • Name (Optional) - This is the label that will show over the transition line in the diagram
  • From State (Required) - A transition connects two states; the "from" state is where the transition begins
  • To State (Required) - A transition connects two states; the "to" state is where the transition ends

Note: The same state may be entered at "From State" and "To State," in which case on the diagram you should see a small loop exiting and re-entering the relevant state

  • Type (Required) - The type will be one of the following:
    • Command - Transition happens when the user pushes a button
    • Auto - A rule (based on a condition) determines when the transition happens
  • User Inputs (Optional for Command transition type; N/A for Auto transition type) - Any approvals or comments needed or allowed from the user are listed here
  • Required Permissions (Optional for Command transition type; N/A for Auto transition type) - If the user to be executing a Command needs a particular permission type, this is listed here
  • Commands (One is required for Command transition type; N/A for Auto transition type) - For a Command type transition, these are things that must be done before the transition can occur. Further detail needed for commands are:
    • Command Name (Required) - A descriptive name such as "Service Control Approve or Reject" should be assigned to the command
    • Parameters - Example parameters for the command are "Approval: Approved" "Approval: Rejected" or "IsEnabled: true"
  • Conditions (Optional for Auto transition type; N/A for Command transition type) - Here is where conditions–if any–are listed that determine when an "Auto" type transition happen (if the transition type is Auto and no conditions are listed, the transition will happen automatically)

More detail about Type, User Inputs, Required Permissions, and Commands is given at headings below.

Conditions

About Conditions

When using Auto transition type, you will generally set one or more conditions. If no condition is set, then the transition will always take place automatically. At runtime, when the condition is satisfied, the transition will take place; if the condition is not satisfied, the condition will not take place.

With respect to multiple conditions specified in a single transition, all of the conditions must be true to complete the condition requirement.

Caution: Auto transitions are the main area where problems can occur in designing a workflow. In order to provide users maximum flexibility in constructing a workflow, error-checking is minimal. Therefore it is possible to create an error such as an endless loop, or have a combination of conditions that contradict each other and thus prevent the transition from ever occurring.

Add Condition

To add a condition,

  1. If not already in the "Transitions" popup, open the scheme in the work area and double-click the desired transition in the diagram. The "Transition" popup will appear.
  2. Activate the plus button under Conditions.

Troubleshooting Tip: The Conditions header will only appear if the transition type is Auto.

  1. If no parameters have been created to select from, you will be prompted to create a parameter, which is a stored variable that can be kept track of over time. To create a parameter,
    1. Activate the Scheme Parameters button on the Workflow editor menu bar. The "Scheme Parameters" popup will appear.
    2. Activate the + Create button.
    3. Enter a name for the parameter in the Name column.
    4. Enter the display name in the next column.
    5. Using the dropdown list in the Type column, select the parameter type (Approve, Boolean, Decimal, Integer, or String) for the variable.
    6. Activate the Save button to save the parameter. (Alternatively, activate Cancel.) Once you have the parameter, you can continue adding a condition.
  1. If parameters exist, you will be able to view the selection using the first dropdown arrow under Conditions.
  2. If the parameter you would like to base the condition on is not listed,
    1. Return to the substeps under Step 2 and create the desired parameter.
    2. In the diagram in the work area, double-click the transition containing the condition to be created.
    3. Activate the dropdown arrow in the first column under Conditions.
  3. From the dropdown list, select the parameter on which the condition is to be based.
  4. In the second column, activate the dropdown arrow and select a comparison operator. For most condition types, this is a choice between = and !=, that is between "equals" vs "not equal to." For the Decimal condition type, additional choices are also available.
  5. In the next column, select a value to be compared against. See details in the substeps below.
    1. For a condition using an Approval type parameter, the comparison values are Undecided, Approved, and Rejected.
    2. For a condition using a Boolean parameter, a checkbox displays in the column. Click on the checkbox to select between checked and unchecked.
    3. For a condition using a Decimal parameter, enter a number to be compared against.
    4. For a condition using an Integer parameter, enter a number to be compared against.
    5. For a condition using a String parameter, enter a string to be compared against.
  1. If you wish to add another condition, activate the plus button, then repeat steps 5 through 7.
  2. Optionally, you can name or rename the auto transition as a whole, under Name near the top of the "Transition" popup.
  3. When you are finished, activate the Close button. The "Transition" popup will close, and you will be returned to the work area.
  4. You may also want to activate the Save icon button on the Workflow editor menu bar at this point, to lock in your work.

Edit Condition

Note: Changes will only apply to future assessments using the edited scheme.

To edit a condition,

  1. In the work area, double-click the transition in the diagram that contains the condition to be edited. The "Transition" popup will appear.
  2. Select an entry under Conditions. (The condition name cannot be edited here. You would have to create a new condition and delete the old one to accomplish a name change.)
  3. Edit the comparison operator as desired. For most condition types, the comparison operator is a choice between = and !=, that is between "equals" vs "not equal to." For the Decimal condition type, additional choices are also available.
  4. As desired, edit the value to be compared against. See details in the substeps below.
    1. For a condition using an Approval type parameter, the comparison values are Undecided, Approved, and Rejected.
    2. For a condition using a Boolean parameter, a checkbox displays in the column. Click on the checkbox to select between checked and unchecked.
    3. For a condition using a Decimal parameter, enter a number to be compared against.
    4. For a condition using an Integer parameter, enter a number to be compared against.
    5. For a condition using a String parameter, enter a string to be compared against.
  1. If you wish to edit another condition, repeat steps 2 through 4.
  2. Optionally, you can name or rename the auto transition as a whole, under Name near the top of the "Transition" popup.
  3. When you are finished, activate the Close button. The "Transition" popup will close, and you will be returned to the work area.
  4. You may also want to activate the Save icon button on the Workflow editor menu bar at this point, to lock in your changes.

  1. In the "Transition" popup, activate the gray "x" at the right of the condition you wish to delete.
  2. WARNING: There will be no confirmation dialog box.

When using Command transition type, you will have at least one Command that will present the user with a button that initiates an action. You may also design in (a) User Inputs and/or (b) one or more Required Permission.

Commands

When you construct a Command transition it will cause a command button to appear on the user's screen at runtime. The text displayed on the button comes from the Command Name field in the popup. You can create more than one command for a transition, which will result in multiple buttons being displayed. Commands effect a transition from one state to another.

A Command transition requires that at least one Command Name be specified in the popup. The initial default Command Name provided in the popup is "Submit."

Delete Command

To delete a command, activate the garbage can icon to the right of the Command Name.

WARNING: Unless the command to be deleted is the last one remaining, there will be no confirmation box displayed.

Tip: A Command transition requires that at least one Command Name be specified in the popup. Therefore if there is only one remaining command and you wish to delete it (rather than just editing it to make desired changes), you would need to create a replacement command first, and only after that delete what had been the last command.

Required Permission

About Required Permissions

If you set up a required permission in your scheme, then when a user adds an assessment based on that scheme, one or more ESMS users must be assigned to each of the permissions roles specified, at the time the assessment is created.

Note: When you use a required permission in a transition specification, you can only specify one required permission per transition. If you want to use more than one required permission, you can make use of multiple self-referencing transitions, or you can consider an alternative approach to collecting approvals.

When you create one or more commands in a transition that also has required permission, the resulting command button(s) will appear in the Workflow area at the bottom of a Assessment screen only for a user with the permissions role specified. For example, if the transition requires PM approval via a required permission in a command type transition, then a user with the Accountant role who is not also a PM will not see the command button that was designed for that transition.

Add Required Permission

To add a required permission,

  1. If not already in the "Transitions" popup, open the scheme in the work area and double-click the desired transition in the diagram. The "Transition" popup will appear.
  2. Activate the dropdown arrow under Required Permission.
  3. Select the permissions role the end user must have in order to activate the transition with the command button.
  4. If you want the user to receive notification (at the bell icon button on the main menu bar), then mark the Notifications enabled checkbox.
  5. It's a good idea at this point to edit the Command Name to make clear from whom the permission is required.
  6. When you are finished, activate the Close button. The "Transition" popup will close, and you will be returned to the work area.
  7. You may also want to activate the Save icon button on the Workflow editor menu bar at this point, to lock in your changes.

Edit Required Permission

Note: Changes will only apply to future assessments using the edited scheme.

To edit a required permission,

  1. In the work area, double-click the transition in the diagram that contains the required permission to be edited. The "Transition" popup will appear.
  2. To change the permission type required, activate the dropdown arrow under Required Permission and select the permission type desired.
  3. To change whether notifications are enabled, check or uncheck the Notifications enabled checkbox.
  4. Change name(s) of the command(s) as desired.
  5. Optionally, you can name or rename the whole transition, under Name near the top of the "Transition" popup.
  6. When you are finished, activate the Close button. The "Transition" popup will close, and you will be returned to the work area.
  7. You may also want to activate the Save icon button on the Workflow editor menu bar at this point, to lock in your changes.

Delete Required Permission

  1. In the work area, double-click the transition in the diagram that contains the required permission to be deleted. The "Transition" popup will appear.
  2. Activate the dropdown arrow under Required Permission.
  3. In the dropdown list, select the empty line at the end.
  4. When you are finished, activate the Close button. The "Transition" popup will close, and you will be returned to the work area.
  5. You may also want to activate the Save icon button on the Workflow editor menu bar at this point, to lock in your changes.

Delete a Transition from a Scheme

To delete a state or transition within a scheme, select the state or transition in the diagram and activate the Delete key on the keyboard. Make sure to save your change(s) by activating the Save icon button on the Workflow editor menu bar.

Note: If you just added the transition you want to delete, simply activating the Delete key is sufficient. However, this delete procedure goes back only one step, to the most recent state or transition you added. Elements placed in the diagram earlier need to first be selected before you activate Delete.
